# Silicon: Tech Tycoon: Audit 2026-07 (Hardening & Polish Pass)

A fresh full-codebase audit run after IMPROVEMENT_PLAN.md completed. Six parallel review agents
swept the 3D subsystem, the pure engine, state/persistence, the React UI, the secondary game
systems, and cross-cutting hygiene. Every CRITICAL/HIGH claim below was hand-verified at the cited
line before it earned a place here.

## Headline (read this first)

**No ship-blocking bugs were found.** The engine and state layers are genuinely clean: determinism
is intact (zero `Math.random`/`Date.now`/`new Date()` in `engine/`), there is no swallowed-error
`catch {}` anywhere, `any` is fully quarantined to the persistence migration block, there is no real
TODO/FIXME/HACK debt, and `tsc -b` is green. This is a **hardening + polish** list, not a
"fix the broken game" list.

Most of the raw agent output graded itself CRITICAL/HIGH; verification knocked the majority down to
LOW or dismissed them (see "Verified non-issues"). What survives is a small, high-confidence set
concentrated in the **3D decoration subsystem** (which matches the working branch) plus a few cheap
infrastructure and robustness wins.

## Working constraints (unchanged from IMPROVEMENT_PLAN.md)
1. One logical change per commit, left committable, with its message.
2. `engine/` stays pure (no React/DOM, no `Math.random`/`Date`, seeded rng only).
3. New persisted fields are OPTIONAL so old saves stay byte-identical (golden invariant).
4. Premium mandate: DesignSystem tokens, 8pt grid, no cramped/blank states.
5. The long-dash character is banned in ADDED lines (code, copy, comments, commits). Scan
   `git diff | grep '^+' | grep -P '\xe2\x80\x94'` before every commit. Pre-existing occurrences are
   NOT churned.
6. Verify every fix: `npm run build` (authoritative typecheck) + `npm test`, and `npm run sim` for
   anything touching balance. Verify 3D changes in a real dark-mode page context.
7. Protected without explicit sign-off: `engine/`, persistence schema + migrations, DeviceRenderer.

## Verified NON-issues (do NOT re-investigate)
The sweep raised these as CRITICAL/HIGH; each was checked and is correct-as-written:
- **WallTV texture "leak"** (`Garage3D.tsx:1322`): `useEffect(() => () => tex.dispose(), [tex])`
  disposes the `CanvasTexture` correctly; the inline `<meshStandardMaterial>` is auto-disposed by
  r3f. No leak.
- **GLTF clone "shared material" bug** (`gltfFurniture.tsx:40`): `scene.clone(true)` shares geometry
  and materials by design; the code deliberately wraps the clone and only transforms the wrapper
  (position/scale), never mutating materials. Safe today; a latent risk ONLY if per-instance
  material tints are later added (captured as item 1.3 below).
- **ContactShadows "per-frame thrash"** (`Garage3D.tsx:~1747`): uses `frames={1}` and re-bakes only
  when its geometry-signature key changes. This is the documented intent, not a leak.
- **Drag "stale coords on unmount"** (`Garage3D.tsx:1446`): `dragIid`/`dragCell` are local state and
  the move only commits on `pointerup`; unmount cancels cleanly. (The real gap here is
  `pointercancel`, item 1.1.)
- **persistence.ts achievement backfill uses "wrong state"** (`:537/:546`): `const s: any = state`
  (`:260`) makes `s` and `state` the SAME reference, so backfills are visible to
  `evaluateAchievements(deriveFacts(state))`. Not a bug.
- **Engine div-by-zero (novelty/factories/market)**: reachable ONLY if a BALANCE constant is set to
  a pathological value (`similarityFloor=1`, `capacityPerWeek=0`, `guidanceFitFloor>1`). Current
  constants (0.78, factory-defined, 0.9) are all safe. Filed as cheap defensive hardening (Phase 3),
  not live bugs.

## PHASE 1: 3D decoration subsystem (the branch's focus, do first)

### 1.1 Furniture drag ignores `pointercancel` (mobile touch interruption leaves item stuck) `[glitch / MED]`
- **Where:** `src/garage3d/Garage3D.tsx:1446-1457` (the `up` listener registers only `pointerup`).
- **Problem:** On touch devices the browser fires `pointercancel` (system gesture, notification, palm
  rejection) INSTEAD of `pointerup`. When that happens mid-drag, the commit-and-reset handler never
  runs, so `dragIid` stays set: the piece is left lifted (y=0.2), following nothing, until the next
  stray `pointerup`. This is a mobile-first game, so the path is realistic.
- **Fix:** register the same handler for `pointercancel` (reset `dragIid`/`dragCell` WITHOUT
  committing the move, so an interrupted drag snaps back). One added `addEventListener` +
  `removeEventListener` pair sharing the `up`/cancel logic.
- **Verify:** manual dark-mode drag; simulate `pointercancel` via devtools; confirm the piece snaps
  back and drag state clears.
- **Effort:** S.

### 1.2 Roaming-staff render cap: hidden coupling to facility capacity `[clarity / LOW]` DONE
- **Where:** `src/garage3d/Garage3D.tsx` (`overflow.slice(podCount, 16)`).
- **Reassessed:** NOT reachable today. Headcount is hard-capped at `facility.staffCapacity` (max 16)
  by BOTH `hireStaff` and `hireCandidate` in `gameState.ts`, and the render cap was also 16, so no
  employee is ever actually dropped. The 3D agent's "hiring beyond ~13 loses positions" was wrong.
- **Real residual (fixed):** the magic `16` was a silent coupling: a future facility tier raising
  `staffCapacity` past 16 would make the cap bite invisibly. Replaced with a named `ROAM_RENDER_CAP`
  constant documenting that it must stay >= the largest facility `staffCapacity`, so the invariant is
  explicit instead of a coincidence. No behavior change (`roamHomeFor` already fans any count out
  safely on golden-angle rings).
- **Effort:** XS (done).

### 1.3 GLTF per-instance material safety (pre-emptive, before any tint feature) `[improvement / LOW]`
- **Where:** `src/garage3d/gltfFurniture.tsx:38-70`.
- **Problem:** Not a bug now, but the shared-material clone means the moment anyone adds a
  per-instance tint/emissive to furniture (a plausible cosmetic feature), every placed copy mutates
  together. This is a tripwire for a future session.
- **Fix:** leave as-is for now; add a one-line code comment flagging "materials are shared: clone
  them before any per-instance material edit." Only implement the deep material clone WHEN a tint
  feature lands.
- **Effort:** XS (comment) now; S later.

### 1.4 3D palette + parametric furniture colors are hardcoded, theme-unaware `[premium / LOW-MED]` WON'T-DO
- **Where:** `src/garage3d/palette.ts`, `src/garage3d/furniture3d.tsx:13-17`.
- **Reassessed as against design intent (not a defect):** `palette.ts` is explicitly documented as
  "fixed character/material colours (intrinsic to objects), chosen to read premium in both light and
  dark UI themes," and `furniture3d.tsx` marks `FABRIC`/`WOOD` as "theme-stable." The furniture ALSO
  already threads `RoomPalette` and keys on dark/light (`p.desk` is warm wood). Making tints track the
  wall-style choice is a taste change that contradicts the stated "premium through restraint"
  philosophy, carries visual-regression risk, and needs a design decision plus a screenshot loop.
- **Decision:** dropped. If the designer later WANTS style-reactive furniture, it is a deliberate
  feature, not a bug fix, and should be scoped with reference mockups. Not doing it speculatively.

## PHASE 2: Infrastructure & enforcement (cheap, high leverage)

- **Problem:** Rules the team assumes are enforced (`react-hooks/exhaustive-deps`, `no-console`) are

- **Verify:** `npm run lint` runs clean (or surfaces a known, triaged list).

### 2.2 Determinism has no regression test for the RNG itself `[test / MED]`
- **Where:** `src/engine/rng.ts` has no `rng.test.ts`.
- **Problem:** Determinism is the load-bearing property of the whole engine (replays, scenarios,
  the sim harness), yet `mulberry32`/`rngFrom` have no direct test pinning the sequence for a known
  seed or asserting `rngState=0` is respected (the `??` vs `||` bug the code comments warn about).
- **Fix:** add `rng.test.ts` that pins the first N outputs for a fixed seed and asserts a `0` state is
  preserved (not treated as falsy).
- **Effort:** S.

### 2.3 Missing tests for `marketing.ts` and `research.ts` `[test / LOW-MED]`
- **Where:** `src/engine/marketing.ts`, `src/engine/research.ts` (no test files).
- **Problem:** Both hold real branching logic (channel selection, RP gates / research forks) and feed

- **Fix:** add focused unit tests for channel affinity/selection and research-fork gating (the fork
  mutual-exclusion is already integration-tested in `state/researchFork.test.ts`; add the pure-unit
  layer).
- **Effort:** S-M.

## PHASE 3: Engine defensive hardening (not reachable today, cheap insurance)

### 3.1 Guard the three misconfig-only division sites `[bug-latent / LOW]`
- **Where:** `novelty.ts:63` (`1 - similarityFloor`), `factories.ts:38` (`/ capacityPerWeek` in the
  `stretch` path), `market.ts` guidance half-width (`Math.sqrt(2*Math.log(1/guidanceFitFloor))`).
- **Problem:** Each divides/logs by a BALANCE-derived value that is safe with today's constants but
  produces `Infinity`/`NaN` if a future tune crosses a boundary (floor -> 1, capacity -> 0, floor
  -> >1). A silent `NaN` here would poison demand/pricing.
- **Fix:** clamp denominators (`Math.max(epsilon, ...)`) or early-return, and add these as cases to
  `balanceGuards.test.ts` so a bad retune fails a test instead of the game.
- **Verify:** `balanceGuards.test.ts` covers the pathological inputs; `npm run sim` unchanged.
- **Effort:** S.

## PHASE 4: Persistence robustness (minor, player-trust)

### 4.1 Quota-trim toast undersells the loss `[polish / LOW]`
- **Where:** `src/state/persistence.ts:~60` (trim toast tone is `neutral`).
- **Problem:** When storage is full the save is trimmed (finished-product sales history dropped) and
  the player is toasted, but with `neutral` tone for what is real (if bounded) data loss.
- **Fix:** escalate the trimmed-save toast to a warning tone and tighten the copy to name what was
  dropped. (Do NOT change what is trimmed: that behavior is correct.)
- **Effort:** XS.

### 4.2 Migration silently drops corrupted `launched` entries `[improvement / LOW]`
- **Where:** `src/state/persistence.ts:~501-503` (filters out products missing finite `price` /
  string `category`).
- **Problem:** A hand-edited/corrupted save loses launch history entries with no signal. This is the
  correct safety behavior (better than crashing), but it is invisible.
- **Fix:** count dropped entries and, if > 0, emit a single `console.warn` and/or a one-time toast so
  the loss is diagnosable. No change to the drop itself.
- **Effort:** XS.

- **Where (worst offenders, verified against `tokens.css`):**
  - `hud.css`: `.hud__pause` 46px (off 8pt), `font-size:12px` (no token; scale is 11/13),
    `.hud__chip` `gap:4px`/`padding:6px 10px` (have `--sp-4/6/10`).

  - `launchReveal.css`: `44px`, `220px`, `gap:5px/3px`, `padding:5px 12px`, a raw `box-shadow`

  - `bottomNav.css`: `gap:3px`, `padding:4px 0`, `top:-2px`, `right:-6px`.
- **Problem:** The token scale is `--sp-2/4/5/6/8/10/12/14/16/24/32/48`; values like `1/3px` and
  `34/44/46px` are genuinely off it. This is polish DEBT (not a bug), but it violates the RULE #1

- **Fix:** a mechanical sweep replacing off-scale px with the nearest token (or promoting a value to a
  new token where a real gap exists, e.g. an icon-glyph size token). Do this as ONE reviewable commit
  per file cluster, verified with before/after screenshots in dark mode so nothing shifts visually.
- **Note:** Do NOT blanket-replace responsive `clamp()` font values or intentional 1px hairlines;
  those are deliberate. Tap targets at 44/48px are HIG-legitimate but should still map to a token.
- **Effort:** M (spread across a few commits).
