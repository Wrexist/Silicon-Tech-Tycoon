## Deferred — prioritized follow-ups

### 1. Native hydration race → total company loss (P1, iOS-native only) — TOP PRIORITY
`src/main.tsx:18` · `src/state/nativeStore.ts:53` · `src/state/useGame.tsx` (boot + autosave).

**Mechanism.** Boot does `Promise.race([hydrateFromNative(), timeout(1200)])`. If the native
Preferences restore takes >1200ms after WKWebView eviction wiped localStorage, the timeout wins, the
app boots a **fresh** `newGame()`, and the ~800ms-debounced autosave writes that fresh game to **both**
localStorage and the Preferences mirror. The still-running `hydrateFromNative` then sees
`localStorage.getItem(key) != null` (now the fresh game) and **skips** the restore — the evicted
company is gone from both stores. The code comment's "restored on the next launch" is false, because
`localStorage wins when both exist` and localStorage now holds the fresh game forever.

**Why deferred, not fixed here.** The only correct fix gates the game-seed read and *all* saving on
actual hydration completion (not the 1200ms timeout) — a change to the boot/save path that (a) is
iOS-native-only, so it cannot be exercised by the web test suite or in this environment, and (b) risks
every user's save if wrong. It must be built and verified on a device. Partial mitigations in
`nativeStore` alone don't work: once the fresh game is in localStorage, the `localStorage-wins` rule
strands the real company in the mirror.

**Fix design.** On native: keep the 1200ms cap for *first paint only*; defer `loadResult()` and
suppress `save()`/`mirrorToNative` until `hydrateFromNative` resolves; if it restores the save key
after the timeout already booted a fresh game, `window.location.reload()` to apply it. Guard the whole
thing on `isNative()` so the web path is byte-identical.

### 2. Other data-integrity edges (P3, quota/replay gated)
- **`trimState` drops `launched.slice(0,12)`** (`persistence.ts:75`) — the quota-fallback save silently
  sheds older launched products (and their remaining revenue). Prefer trimming `cashHistory`/`feed`
  first; only shed `launched` as a last resort, and toast when it happens.
- **Museum key collision on fixed-seed replays** (`useGame.tsx:746`, `museum.ts:62`) — replaying a
  daily/challenge seed and shipping the same product at the same week overwrites the prior enshrinement
  (`${seed}-${id}-${week}` collides). Design call: append a monotonic launch counter *if* duplicate
  enshrinements are wanted.

### 3. Performance (P2–P3, not on the always-mounted hot path)
- **Profile-store read cache** (`scenarioProgress.ts` / `challengeProgress.ts`) — `readMasteryInput`
  parses localStorage every tick, and `ChallengeTracker`/`ScenarioTracker` re-parse per render during a
  run. A module-level cache is the obvious fix **but** the integration tests swap `localStorage` in
  `beforeEach` without `vi.resetModules()`, so a module cache would leak stale data across tests. Do it
  with either a test-visible reset hook or per-component `useMemo`, not a bare module cache.
- **Unmemoized per-tick derivations** — `StrategicInsightsCard`/`PerformanceCard` (`HQ.tsx`, always
  mounted), `Market.tsx:172`, `Company.tsx:142`: wrap each view-model in `useMemo` keyed on the inputs
  it reads. Low individual cost, but on the constant render path.
- **`Factory3D.tsx:1139`** per-frame array/tuple allocations in `useFrame` (factory world only) — mutate
  in place / reuse a scratch tuple.

### 4. Design polish (P3, visual-review needed — deliberately not auto-applied)
- `.ds-sheet-scrim` (`primitives.css:363`) and `.dtut-scrim` (`decorateTutorial.css:12`) use
  `rgba(10,12,16,0.3)` instead of the house `color-mix(in srgb, var(--bg) 30%, transparent)` scrim;
  converting changes the light-theme dim, so it wants a design eye.
- `company.css:195` mixes a literal navy `#1e3a8a`; needs a token whose value matches or a deliberate
  re-choice.
- Raw `★` glyph as a rating icon (Scenarios/Progress/ScenarioTracker/ResultCard) vs Lucide `<Star>` —
  borderline (typographic, not emoji); consistency call.

### 5. A11y + test gaps (P3)
